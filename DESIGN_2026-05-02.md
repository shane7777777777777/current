## Non-Negotiables Embedded in Every Phase

These are stated up front because they recur in every phase's deliverables and success criteria. If a deliverable in any phase below conflicts with one of these, the deliverable is wrong.

1. **No silent writes. Ever.** Every mutation to any external system OR any Phoenix CRM table is logged with `(actor, target_system, target_record, before_state, after_state, reason, correlation_id, timestamp)`. There is no "log-disabled" mode. There is no "skip log on retry" path. The logger failing is a hard error that aborts the write.
2. **Full audit trail.** Append-only, immutable at the storage layer (DB role lacks UPDATE/DELETE on the audit table), queryable by `(actor, time-range, target_system, correlation_id)`. The audit table is the second non-negotiable; everything else is layered on top of it.
3. **Dry-run before live mutation.** Default mode for every write tool is `preview` — returns the exact payload that *would* be sent, the affected records, and the diff. `commit` requires an explicit second call with the preview's correlation_id as proof-of-review. (The current PCS `SF_APPROVAL_TOKEN` / `ALLOW_SF_WRITES` gate is a coarser version of this; the successor formalizes it.)
4. **Service Fusion remains operational** until the replacement is **proven** per Phase D criteria. No half-cutovers. No "we'll just stop using SF for new customers" — that creates a divergent state with no rollback.
5. **Design-review checkpoint before any Phase C implementation.** This document, post-revision-with-Shane's-input, is the gate. Phase C does not start until Shane signs off on Phase B.
6. **Phoenix Electric is an ELECTRICAL company** (not HVAC/plumbing). Code prefixes are **SW_, LT_, CKT_** (NOT PE_NC_). Domain language used by the system reflects this — "circuits", "panels", "loads", "load calculations", not generic "service items."
7. **Read-only on Microsoft Graph and ServiceTitan during the entire scope of this design.** PCS already obeys this; the successor inherits it. Writes against those systems are out of scope and require a separate design.
8. **No deletion ever.** Soft-delete with `archived_at` and `archived_by` columns is fine. Hard `DELETE` is not.

## Phase B — Phoenix CRM Successor (Design Only)

**Goal:** Produce the architectural blueprint for the SF replacement. No code. Output is a set of contracts, schemas, and flow diagrams that Phase C executes.

This phase is by far the largest in this document because it's where the foundational decisions land. It splits into five sub-deliverables: data model, audit trail, write controls, UX flows, migration path. Each has its own scope, success criteria, and open questions.

### B1 — Data Model

**Scope:** A relational schema covering the eight CRM domains (customers, jobs, estimates, invoices, scheduling, pricebook, reporting, history) plus the cross-cutting domains (audit, identity, configuration). The schema is normalized where SF is normalized, denormalized where SF returns nested objects we don't want to re-pivot every read.

**Domain breakdown** (each gets its own ERD section in the design output):

1. **Customers** — `customers`, `customer_contacts` (1:N), `customer_locations` (1:N), `customer_tags` (M:N), `customer_custom_fields` (EAV or JSONB — recommend JSONB for flexibility, validated against a schema-of-schemas table). SF maps cleanly.
2. **Jobs** — `jobs`, `job_appointments` (1:N), `job_techs_assigned` (M:N), `job_notes` (1:N append-only), `job_documents` (1:N), `job_signatures` (1:N), `job_visits` (1:N), `job_status` (enum referencing `job_statuses` lookup). Job has 49 fields and 20 expandable arrays in SF — Phase B decides which are first-class columns vs JSONB.
3. **Estimates** — `estimates`, `estimate_line_items` (1:N — products, services, labor, expenses, other_charges as separate tables OR one polymorphic `line_items` table with `kind` discriminator; recommend polymorphic for queryability), `estimate_status` (enum). The line-item structure is where Shane's three-tier markup rules attach.
4. **Invoices** — `invoices`, `invoice_line_items` (1:N), `invoice_payments` (1:N), `invoice_status` (enum). Read-only via SF API but writable in Phoenix CRM post-cutover.
5. **Scheduling** — `calendar_tasks`, `appointments` (links to `jobs` and `technicians`), `technician_availability` (windows), `routes` (sequence of appointments per tech per day, with GPS waypoints). SF only exposes calendar-tasks; the rest is greenfield in Phoenix CRM.
6. **Pricebook** — `services`, `materials`, `equipment`, `categories` (hierarchical), `vendor_prices` (M:N from materials to vendors with effective dates), `markup_rules` (the three-tier Premium/Member/List rules). Entirely greenfield (SF API has none of this).
7. **Reporting** — Not its own table but a set of materialized views: `mv_jobs_by_status_by_week`, `mv_estimate_pipeline`, `mv_invoice_aging`, `mv_tech_utilization`, `mv_revenue_by_source`. Refreshed on a schedule (start with hourly, tune later).
8. **History** — Two distinct things: (a) the `audit_log` table (see B2), which is the *transactional* history; (b) `entity_history` views that project the audit log into "show me everything that happened to customer X" and "show me job Y's full timeline."

**Cross-cutting tables:**

- `users` — Phoenix CRM users (keyed to M365 identity per DEC-5).
- `roles`, `permissions`, `user_roles` — RBAC primitives. See B3.
- `audit_log` — see B2.
- `external_id_map` — `(local_table, local_id, external_system, external_id)` — every Phoenix CRM record carries a back-reference to its SF counterpart during Phase C. After cutover, this becomes the historical link.
- `sync_state` — per-table `(last_sync_at, last_sync_token, status)` for the Phase C mirror.
- `correlation_log` — every inbound request gets a UUID; every downstream call carries it. AsyncLocalStorage already scaffolded in `packages/shared/src/logger.ts:8`.
- `dry_run_previews` — TTL'd table holding preview results so the commit call can verify "this is exactly what was previewed." See B3.

**Naming conventions:**

- `snake_case` table and column names (Postgres convention, matches SF's API field names).
- Primary keys: `id BIGSERIAL` for high-volume tables (jobs, line items, audit), `UUID` for everything else and for any record that crosses system boundaries.
- Timestamps: `created_at`, `updated_at`, `archived_at` (soft-delete), all `TIMESTAMPTZ`. No `DELETE` ever (non-negotiable #8).
- Money: `NUMERIC(12, 2)`, never `FLOAT`. Currency is implicit USD; add `currency_code` if Phoenix ever expands.

**Success criteria for B1:**

- Every SF v1 GET endpoint that PCS exposes (customers, jobs, estimates, invoices, techs, calendar-tasks, lookups) maps cleanly to one or more Phoenix CRM tables, with JSONB columns where the SF nesting doesn't fit relational.
- The pricebook schema captures Shane's three-tier markup rules (Premium +20% / Member +30% / List +40%) as configurable rules, not hardcoded constants.
- Every table has `id`, `created_at`, `updated_at`, `archived_at`, `created_by`, `updated_by` (or a documented exception per table).
- Domain language reflects electrical contracting: line items in estimates can carry `circuit_id`, `panel_id`, `load_amps`, `breaker_size`, etc. — not generic `widget_type`.
- ERD checked into the design repo (likely as Mermaid in the design doc + a generated PDF).

**Open questions (B1):**

- **DEC-B1.1:** EAV vs JSONB for SF's `_expandable` and custom_fields? Recommend JSONB with a `custom_field_schemas` table that defines the validators. EAV makes queries painful.
- **DEC-B1.2:** Polymorphic `line_items` table or separate `products`/`services`/`labor_charges`/`expenses`/`other_charges`? SF separates them; Phoenix CRM could collapse for query simplicity. Recommend polymorphic with a `kind` discriminator + JSONB `kind_specific` column.
- **DEC-B1.3:** ID strategy at cutover — keep SF integer IDs as-is in `external_id_map`, generate fresh Phoenix CRM IDs (BIGSERIAL or UUID), or attempt to *reuse* SF IDs as Phoenix CRM IDs? Recommend fresh IDs + map. Reusing SF IDs sounds clever and creates collisions later.

### B2 — Audit Trail

**Scope:** A single, append-only, immutable log of every mutation across every domain. This is the most important table in the system; everything else is layered on top.

**Schema:**

```sql
CREATE TABLE audit_log (
  id              BIGSERIAL PRIMARY KEY,
  occurred_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
  correlation_id  UUID NOT NULL,
  actor_id        UUID NOT NULL REFERENCES users(id),
  actor_kind      TEXT NOT NULL,  -- 'human' | 'agent' | 'system' | 'migration'
  action          TEXT NOT NULL,  -- 'create' | 'update' | 'archive' | 'external_call' | 'external_response' | 'business_event'
  target_system   TEXT NOT NULL,  -- 'phoenix_crm' | 'service_fusion' | 'graph' | 'quickbooks' | 'rexel'
  target_table    TEXT,           -- nullable for external systems
  target_id       TEXT,           -- string for cross-system compat (SF uses int, Phoenix uses UUID)
  before_state    JSONB,          -- null for create, full record for update/archive
  after_state     JSONB,          -- null for archive/external_response, full for create/update
  diff            JSONB,          -- computed: keys that changed, with old/new
  reason          TEXT,           -- human or agent's stated reason
  request_payload JSONB,          -- for external_call: what we sent
  response_payload JSONB,         -- for external_response: what we got
  status          TEXT,           -- 'success' | 'failure'
  error_message   TEXT,           -- if status = 'failure'
  duration_ms     INTEGER,        -- for external_call
  ip_address      INET,
  user_agent      TEXT,
  source          TEXT            -- 'cli' | 'web' | 'mcp' | 'agent' | 'cron' | 'migration'
);

-- Append-only enforcement (DB-level, not application-level)
REVOKE UPDATE, DELETE, TRUNCATE ON audit_log FROM phoenix_crm_app;
GRANT INSERT, SELECT ON audit_log TO phoenix_crm_app;

-- Indexes for the queries we'll actually run
CREATE INDEX idx_audit_correlation ON audit_log(correlation_id);
CREATE INDEX idx_audit_actor_time ON audit_log(actor_id, occurred_at DESC);
CREATE INDEX idx_audit_target ON audit_log(target_system, target_table, target_id);
CREATE INDEX idx_audit_action_time ON audit_log(action, occurred_at DESC);
CREATE INDEX idx_audit_failures ON audit_log(occurred_at DESC) WHERE status = 'failure';
```

**Three layers logged (per DEC-11):**

1. **External API calls.** Every call to SF, Graph, QuickBooks, Rexel logs two rows: `external_call` (with `request_payload`) and `external_response` (with `response_payload` and `duration_ms`). Same `correlation_id` ties them. PCS today logs at info-level via pino but nothing durable.
2. **Internal Phoenix CRM writes.** Every INSERT/UPDATE (no DELETEs) on a non-audit table logs an `audit_log` row with `before_state`, `after_state`, `diff`. Implemented as DB triggers OR application-level wrapper — recommend DB triggers because they can't be bypassed by a forgotten code path.
3. **Business events.** "Estimate sold," "Job completed," "Customer archived," "Markup rule changed." Emitted explicitly by application code. These are the rows humans read.

**Queryability:**

- `WHERE correlation_id = ?` — full request trace across systems.
- `WHERE actor_id = ? AND occurred_at >= now() - interval '7 days'` — what did Stephanie do this week?
- `WHERE target_system = 'service_fusion' AND status = 'failure'` — what's broken between Phoenix CRM and SF?
- `WHERE target_table = 'estimates' AND target_id = 'X'` — full history of estimate X.

**Retention:** Forever. Non-negotiable #8. If volume becomes a problem, partition by month, but never drop.

**Success criteria for B2:**

- Schema reviewed and approved.
- DB role permissions confirmed: `phoenix_crm_app` cannot UPDATE or DELETE audit rows. Only a separate `phoenix_crm_admin` role (used only for schema migrations) has those privileges, and even then audit rows are protected by triggers that reject UPDATE/DELETE attempts with a logged error.
- Every B1 table that holds business data has triggers writing to `audit_log` on INSERT/UPDATE.
- Three layered queries above can be answered in < 100ms on a year of data.
- Migration plan in place to replay PCS's pino logs into the audit trail at Phase C start (so we don't lose the historical PCS write history).

**Open questions (B2):**

- **DEC-B2.2:** PII redaction in `before_state`/`after_state`? Customer phone/email/address are PII. Recommend store as-is; restrict read access via RBAC at query time. Redaction-at-write loses information you might need for forensics.
- **DEC-B2.3:** Replay PCS pino logs into audit trail at Phase C start? Recommend yes for write events only (creates), no for reads. Reads are too noisy and not historically valuable.

### B3 — Write Controls

**Scope:** RBAC, dry-run-by-default semantics, explicit-confirm-to-commit pattern, multi-step approval for high-impact actions.

**RBAC model:**

- **Roles:** `viewer` (read-only), `editor` (create + update non-financial), `finance` (editor + invoice/payment writes), `admin` (everything + RBAC management), `agent` (programmatic, scoped per-agent).
- **Permissions:** Verb on resource — `customers:read`, `customers:write`, `customers:archive`, `estimates:write`, `estimates:sell` (a privileged action), `pricebook:write`, `audit:read`, `users:manage`, etc.
- **Phoenix Electric humans:** Shane = `admin`; Stephanie = `editor` + `finance`; Ash = `editor`; future office staff = `editor` or `viewer` per role.
- **Agents:** Each of the 10 Phoenix agents (ServiceTitan Director, Phoenix Courier, Finance Analyst, Pricing Analyst, Quote Generator, Schedule Coordinator, Security Sentinel, Knowledge Keeper, Code Reviewer, Compliance Auditor) gets its own scoped role. Quote Generator can `estimates:write` but not `estimates:sell`. Pricing Analyst can `pricebook:read` but not `pricebook:write`. Etc. Principle of least privilege.

**Dry-run-by-default pattern:**

Every write tool exposes two modes: `preview` and `commit`.

```
POST /v1/estimates  { mode: 'preview', ...payload }
→ 200 OK
{
  preview_id: 'uuid-1',
  expires_at: '2026-05-02T19:30:00Z',  // 5 min TTL
  would_create: { id: null, ...computed_estimate },
  would_affect: ['customer:123', 'job:456'],
  would_call_external: ['service_fusion: POST /v1/estimates'],
  diff: null,  // no existing record for create
  warnings: ['Markup tier set to List (+40%); customer is tagged as Member (+30%) — confirm intent'],
  required_role: 'editor'
}

POST /v1/estimates  { mode: 'commit', preview_id: 'uuid-1' }
→ 201 Created
{ id: 'uuid-7', ... }
```

The commit call MUST reference a valid, non-expired `preview_id` that was generated by the same actor. This is enforced at the API layer, not by convention. The `dry_run_previews` table TTLs its rows at 5 minutes.

This subsumes and replaces PCS's `SF_APPROVAL_TOKEN` / `ALLOW_SF_WRITES` env-flag pattern. It's stricter (per-call, not per-process) and more auditable (preview AND commit both log).

**Multi-step approval (privileged actions):**

Some actions require a second human's approval before commit, even for `admin`:

- `estimates:sell` (commits an estimate to invoice — legal/financial impact)
- `pricebook:bulk_update` (more than N items at once)
- `users:manage` (granting/revoking roles)
- Any write where Phoenix CRM and SF are about to disagree (Phase C reconciliation conflict)

The pattern: commit returns `202 Accepted, awaiting approval`, writes a row to `pending_approvals`, notifies the second-approver via Teams/email/Telegram, and only completes when the second approval arrives. Audit log shows both actors.

**Success criteria for B3:**

- Every write tool in Phase C has `preview` and `commit` modes; no write tool accepts a single call that mutates state.
- RBAC matrix written out: every (role, resource, action) cell is defined.
- The Phase A `logApproval()` hook becomes the audit-write call in Phase B/C.
- A "policy as code" file (e.g. `policies.yaml`) defines what's privileged; changing the policy is itself an audited admin action.

**Open questions (B3):**

- **DEC-B3.1:** Where does the second-approver notification go? Recommend Microsoft Teams (Phoenix already has it) with a magic-link approve/reject. Telegram is fine for Shane personally but not extensible to staff.
- **DEC-B3.2:** Preview TTL — 5 min, 30 min, longer? 5 min is tight; 30 min may stale. Recommend 30 min default, configurable per-action (estimates: 30 min; high-impact like `pricebook:bulk_update`: 5 min).
- **DEC-B3.3:** Can agents (programmatic actors) commit without preview, given that their inputs are deterministic? Recommend NO — agents do `preview` first, log it, then `commit`. The deterministic-input argument is exactly the kind of "we don't need the safety here" reasoning that creates the bypass paths that bite later.

### B5 — Migration Path

**Scope:** How data moves out of SF and into Phoenix CRM during Phase C, and how the cutover happens in Phase D. Per DEC-2 the recommendation is **read-only mirror with writes-target-SF-only during Phase C**, and per DEC-9 the recommendation at cutover is **rolling backfill + SF kept read-only forever**.

**Mirror strategy (Phase C):**

- **Initial backfill:** One-time full pull of every record from every SF endpoint that PCS exposes. Stored in Phoenix CRM tables with `external_id_map` rows pointing back to SF IDs. Run via a dedicated migration tool, not the normal application path. Backfill is the only time `actor_kind = 'migration'` appears in `audit_log`.
- **Incremental sync:** Every N minutes (start at 15 min, tune), poll each SF endpoint with a cursor (last `modified_on` per table). Upsert changes into Phoenix CRM. Each change writes to `audit_log` with `actor_kind = 'system'`, `source = 'cron'`. Conflicts are impossible because Phase C doesn't accept Phoenix-CRM-side writes.
- **Sync state:** `sync_state` table tracks `(table, last_sync_at, last_modified_seen, status, error)`. Health check exposes lag per table.
- **Failure mode:** If incremental sync fails, the alert goes to Shane via the same Teams/email channel as approval requests. The mirror staling for an hour is a Yellow event; staling for 4+ hours is Red.

**Conflict resolution: not applicable in Phase C.** Per DEC-2, Phoenix CRM doesn't accept its own writes, so it can't disagree with SF. This is by design. Conflicts only become possible at cutover, and at cutover SF becomes read-only — so conflicts STILL aren't possible because there's only one writer.

**Cutover (Phase D — covered in detail below):**

- SF goes read-only at T-0.
- Phoenix CRM begins accepting writes (its own write paths flip from rejected to accepted).
- Final reconciliation pass confirms parity.
- A "writes-disabled in SF" verification: try a write to SF via PCS, confirm it's rejected by SF (~~you can revoke the OAuth client_secret in the SF admin to enforce this~~). **⚠️ UNSAFE — see v2 § "How SF goes read-only without losing read access". Revoking the only client_secret kills reads too.**

**Rollback story:**

- During Phase C: nothing to roll back. SF is still SoR; Phoenix CRM is a derived view.
- At cutover: if Phoenix CRM fails post-cutover, ~~re-enable SF writes (restore the client_secret)~~, point the agents/UI back at PCS, and resume. Phoenix CRM remains a read-only historical mirror until the issue is fixed and a second cutover attempted. **⚠️ UNSAFE — see v2 Phase D Rollback. The client_secret is never revoked in v2, so there's nothing to restore.**
- Post-cutover (no SF writes): SF is the historical archive forever (per DEC-9). If a record is needed and Phoenix CRM lost it, SF has it.

**Success criteria for B5:**

- Every SF endpoint PCS exposes has a documented sync strategy (cursor field, poll interval, error handling).
- The migration tool has a written design before any code is written.
- Reconciliation queries are designed: "for every customer in SF, is there one in Phoenix CRM with matching ID via `external_id_map` and matching key fields (name, primary phone, email)?" Same for jobs, estimates, invoices.
- A "drift detection" job runs nightly and reports any mismatch between SF and Phoenix CRM.

**Open questions (B5):**

Options: live without them, screen-scrape, or import from SF's CSV exports (Shane can run these from the SF admin). Recommend CSV export for one-time historical archive; live without the rest.

## Phase D — Cutover

**Goal:** Define what "Phoenix CRM is proven" means, and the procedure for flipping SF from system-of-record to read-only archive.

This phase is design-only at this point — there is no cutover plan to execute until Phase C ships all six slices and runs cleanly for 90 days. What follows is the *acceptance criteria* and *procedure outline*. The actual cutover plan is written in detail just before cutover, when we know what the system actually looks like.

### "Proven" criteria (all must hold)

1. **Functional completeness.** Every B4 flow works end-to-end in Phoenix CRM. Stephanie has gone an entire week using only Phoenix CRM (with SF tab closed) and reports no missing capabilities.
2. **Data parity.** Nightly drift detection has reported < 0.05% drift for 60 consecutive days. Random spot-check of 100 customers, 100 jobs, 100 estimates, 100 invoices shows all fields match between SF and Phoenix CRM.
3. **Audit trail proven useful.** A real forensics question has been answered using `audit_log` queries (could be as simple as "who changed customer X's address last Tuesday?"). The audit infrastructure is exercised, not theoretical.
4. **Performance under realistic load.** 100 concurrent users (more than Phoenix has, deliberately) sustained for 1 hour without degradation.
5. **Backup and restore tested.** Postgres backup taken, restored to a fresh instance, verified. Audit log preserved through restore.
6. **Security review passed.** External or internal review of RBAC, write controls, secrets handling, audit trail integrity. Not a checkbox — a real review with findings addressed.
7. **Stephanie, Ash, and Shane all sign off.** Each one independently. If any of them isn't ready, cutover doesn't happen.
8. **Field-tech-tooling design exists** (per DEC-10, post-cutover, but the design must exist so techs aren't surprised).

### Cutover procedure (outline — detail at cutover-time)

**T-7 days — Communication:**
- Notify Phoenix Electric staff: cutover date, what changes, who to ask.
- Final dry-run cutover in a staging environment.

**T-1 day — Pre-flight:**
- Last full sync from SF.
- Drift check. Must be zero, not "low" — zero.
- Backup Phoenix CRM database.
- Backup SF data via CSV export (one final snapshot for the historical archive).

**T-0 — Cutover (target window: Friday evening, low-activity):**
1. Announce maintenance window (30 min planned).
2. Stop the incremental sync job.
3. Run one final delta sync.
4. Run drift detection. Zero drift required.
5. ~~Revoke SF OAuth client_secret in SF admin (this disables PCS write tools — the read tools still work because they re-authenticate on each call and will fail; that's intentional).~~ **⚠️ UNSAFE AND SELF-CONTRADICTORY — DO NOT EXECUTE. The original v1 sentence even contradicts itself ("read tools still work" AND "will fail"). Revoking the only `client_secret` disables ALL API access, killing the read path that DEC-D1/D2/D3 and step 8 below depend on. See v2 Phase D step 5 (5a/5b/5c) for the correct layered approach: deploy a cutover PCS image with `ALLOW_SF_WRITES=false` and `create_*` tools physically removed from the registry, then optionally apply a SF-side read-only OAuth role if SF admin supports it. The `client_secret` is preserved.**
6. Flip Phoenix CRM write paths from "reject" to "accept."
7. Update agent config: ServiceTitan Director, Phoenix Courier, Finance Analyst etc. now hit Phoenix CRM endpoints, not PCS.
8. PCS stays installed but `ALLOW_SF_WRITES` is hardcoded to false; the plugin remains as a read-only window into the SF historical archive.
9. Monitor for 4 hours. Anything weird → see Rollback below.
10. Announce completion.

**T+1 day:** Daily standup specifically focused on cutover issues. Stephanie's morning workflow under the new system. Shane reviews audit log volume and content.

**T+7 days:** Retrospective. What surprised us? What broke? Updates to runbook.

### Rollback (if Phase D fails)

If within 48 hours of cutover something is broken enough to require rollback:

1. ~~Restore SF OAuth client_secret. SF write path through PCS resumes.~~ **⚠️ UNSAFE — see v2 Phase D Rollback step 1. v2 keeps the `client_secret` intact across cutover, so rollback is "re-deploy the rollback PCS image (with `create_*` tools intact and `ALLOW_SF_WRITES=true` available)" — no secret to restore.**
2. Flip Phoenix CRM write paths back to "reject." Phoenix CRM resumes mirror mode.
3. Diff the writes that landed in Phoenix CRM during the cutover window. Reverse-apply them as SF writes (manually, with two-person approval per non-negotiable #3, because these are now retroactive writes).
4. Communicate. Schedule a second cutover attempt only after the original blocking issue is fixed AND a new dry-run passes.

After 48 hours, rollback gets harder because real Phoenix CRM-only writes accumulate. After 7 days, rollback effectively means "double-write SF and CRM to converge them," which is its own design problem. The 48-hour window is the safety zone.

### Phase D dependencies

- Phase C complete with 90 days clean.
- All eleven proven-criteria items above.
- Shane's calendar: cutover takes 30 min planned + 4 hours monitoring; do not cutover during any active business event (no Friday-of-major-Generac-launch, no end-of-month-billing-week).
