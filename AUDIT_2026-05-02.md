# AUDIT — Phoenix Current Software (PCS)

**Date:** 2026-05-02
**Auditor:** Claude Opus 4.7 (1M ctx) on behalf of Shane Warehime
**Repo:** https://github.com/GIT-PHOENIX-HUB/phoenix-current-software
**Branch audited:** `main` @ `74c1598` (commit dated 2026-04-05, the most recent on the remote)
**Mode:** Read-only documentation pass. No code changes, no SF/Graph/ST writes triggered.
**Scope:** Step 1 of 4 (scrub). Output is documentation only — design and build come later.

---

## TL;DR (5 bullets)

1. **PCS is a thin, mostly-correct read-mostly Service Fusion v1 wrapper exposed as an MCP server + Claude Code plugin.** The core HTTP client, OAuth, token cache, rate limiter, and 60s/300s/600s response cache are well-shaped and worth preserving. (~1,300 LOC of TypeScript, sane separation.)
2. **The plugin is unrunnable as committed.** `plugin/.mcp.json` points at a hardcoded path inside the dead `phoenix-ai-core-staging` repo, no root `package.json` exists (so `npm workspaces` claims in PRODUCT_BIBLE are aspirational), and `dist/` is not in source. From a clean clone, `npm run build` will fail before the path bug even matters.
3. **Documentation is on three different versions of reality.** Active-tool count is reported as 23 / 21 / 23 (README vs `package.json` vs SKILL.md); deprecated-stub count is 35 / 33 / 34 / 34 (README vs VERIFICATION vs SKILL vs actual code = 34); rate limit is 60 / 60 / 60 / 120 (code vs api-surface vs SKILL vs rewrite-brief). Several skill workflows and one slash command (`/sf-customers`) call deprecated stubs and the wrong tool prefix (`sf_*` vs `servicefusion_*`).
4. **Phoenix-domain rules are mostly honored but with leaks.** The agent definition correctly asserts "ELECTRICAL company (NOT HVAC)". But: `references/future-gateway.md` uses `America/Phoenix` timezone (Phoenix Electric is in **Denver, CO** — `America/Denver`); the agent file itself says "Mountain Time" and conflates "Phoenix" the company with the AZ city in one place. Code prefix conventions (SW_, LT_, CKT_) are not present anywhere in this codebase — neither honored nor violated, since this layer doesn't touch them.
5. **No tests, no CI, no observability, no `.gitignore`, no error budget, no operational story beyond "Shane runs it on his MacBook with `az login`".** The 17,929-line API reference and 4.2 MB RAML spec are checked in but never consumed by code or tests. **Recommendation: KEEP the client/cache/rate-limiter, REPLACE the plugin wiring + tools surface, ARCHIVE the aspirational skill references and rewrite brief.**

### Request flow (read path — happy)

```
1. User: "list customers named warehime"
2. Skill triggers → agent picks tool → MCP CallTool("servicefusion_search_customers", {...})
3. Server validates with zod, dispatches to tool.handler
4. handler → client.get("/v1/customers", {filters[customer_name]: "warehime"})
5. client.request():
   a. ensureToken() — getToken() reads Key Vault if first call, otherwise cached
   b. cache.get(path, params) — return early on hit
   c. rateLimiter.acquire() — block if remaining ≤ 0
   d. fetch(SF API) with Bearer token
   e. updateFromHeaders(X-Rate-Limit-*)
   f. on 429 → handle429(), wait, retry once
   g. on 200 → cache.set() and return JSON
6. Server JSON.stringify(result) → MCP CallToolResult.content[0].text
```

### Write path

Identical, but `requiresApproval: true` triggers a check at [client.ts:161](packages/mcp-server/src/client.ts:161): write blocked unless `SF_APPROVAL_TOKEN` is set or `ALLOW_SF_WRITES=true`. After 200, cache for that path prefix is invalidated ([client.ts:251](packages/mcp-server/src/client.ts:251)). This is good — it means staleness window for create-then-list is bounded by write-path invalidation, not just TTL.

### Transport modes

- **stdio** (default) — used by Claude Code plugin. CLI mode entered via `main()` at [index.ts:188](packages/mcp-server/src/index.ts:188).
- **http** — `StreamableHTTPServerTransport` mounted on Express at `POST/GET /mcp`. Health probe at `GET /health`. Bound to `127.0.0.1:3100` by default. **No auth in front of the HTTP transport** — see §5.

## 8. Open Questions for Shane (blockers for clean redesign)

These are the calls only you can make. None of them have an obvious right answer from the code.

1. **Plugin co-location.** `BUILD_DOC.md` line 80 already flagged this: should the `plugin/` directory live in this repo (current state), in a separate `service-fusion-plugin` repo, or be co-located with the built `dist/`? The successor's plugin-install story depends on this answer.
2. **Deprecated stubs disposition.** Keep them as throwing stubs (today), replace with one informational tool, or drop entirely and let the agent definition handle "use the web UI" guidance? My lean is: **drop entirely**, but you may want them as a debugging aid.
3. **Successor scope.** Is PCS staying as a thin SF v1 wrapper for as long as SF is the system of record, or is it being absorbed into a broader Phoenix data plane (with QuickBooks, Graph, ServiceTitan all living behind one MCP)? The shared `keyvault.ts` already has accessors for Graph/SharePoint/Anthropic/OpenAI secrets — was that aspirational or is consolidation imminent?
4. **HTTP transport consumers.** `BUILD_DOC.md` objective 4 is "wire HTTP to Gateway." Is the Gateway running anywhere yet, or is this still on the to-do? Determines whether HTTP transport is a near-term priority or can be deferred.
5. **Test harness against SF.** Mocked fixtures only, live integration against the production tenant, or a dedicated SF sandbox tenant? **Live tests against production are not safe** for write tools — even with the approval gate, a flaky test could create test customers in your real CRM. (`BUILD_DOC.md:86` flagged this; still unanswered.)
6. **Tool naming convention.** `servicefusion_*` (current code) vs. `sf_*` (per the rewrite brief and most skill docs) vs. just unprefixed (relying on the MCP namespace `mcp__servicefusion__`). Pick one and enforce.
7. **References bloat.** Is the 4.2 MB RAML + 200 KB scrape + 17,929-line markdown actually used by anyone, or were they checked in "in case we need them"? My recommendation is to move them out of the repo, but you may have tooling that reads them.
8. **Approval/audit story.** `logApproval()` exists and is uncalled. What level of audit trail do you want for SF writes — pino log line, append to a SharePoint/Teams audit channel, or full SOX-grade ledger? Depends on whether PCS is in-scope for the broader Phoenix governance work.
9. **Successor stack.** Stay TypeScript/MCP-SDK, or go Python (FastMCP / MCP-Python), or go a managed agent runtime? The current TS code is fine; the question is whether the successor gets folded into the broader Phoenix-AI stack which may have a different language preference.
10. **Vault secret naming canonicalization.** `keyvault.ts` tries 3–6 names per secret. Pick the canonical name in vault, document it once, and remove the fallbacks (or keep the fallbacks but log warnings). The current "tries everything" is robust but obscures which name is real.
