## Non-Negotiables Embedded in Every Phase

(Unchanged from v1. Restated for completeness.)

1. **No silent writes. Ever.** Every mutation to any external system OR any Phoenix CRM table is logged with `(actor, target_system, target_record, before_state, after_state, reason, correlation_id, timestamp)`. There is no "log-disabled" mode. The logger failing is a hard error that aborts the write.
2. **Full audit trail.** Append-only, immutable at the storage layer (`audit_log` role lacks UPDATE/DELETE), queryable by `(actor, time-range, target_system, correlation_id)`.
3. **Dry-run before live mutation.** Default mode for every write tool is `preview`. `commit` requires an explicit second call referencing the preview's `preview_id`. PCS's `SF_APPROVAL_TOKEN` / `ALLOW_SF_WRITES` env-flag pattern is the v0 of this; the successor formalizes it per B3.
4. **Service Fusion remains operational** until the replacement is **proven** per Phase D criteria.
5. **Design-review checkpoint before any Phase C implementation.** This v2 doc, post-Shane-final-lock, is the gate.
6. **Phoenix Electric is an ELECTRICAL company** (not HVAC/plumbing). Code prefixes are **SW_, LT_, CKT_** (NOT PE_NC_). Domain language reflects this: circuits, panels, loads, load calculations.
7. **Read-only on Microsoft Graph during the entire scope of this design.** (ServiceTitan removed — see § Removed From Scope below.)
8. **No deletion ever.** Soft-delete with `archived_at` and `archived_by`. Hard `DELETE` is not allowed.

## Phase B — Phoenix CRM Successor (Design Only)

**Status:** Direction approved 2026-05-02. v2 deepens B1–B5 with locked answers. Awaits one more review for final lock.

**Goal:** Architectural blueprint for the SF replacement. No code. Output is contracts, schemas, and flow diagrams that Phase C executes.

### B1 — Data Model

Per DEC-3 the schema is Postgres. Per DEC-B1.1, custom fields are JSONB validated by `custom_field_schemas`. Per DEC-B1.2, line items are polymorphic with a `kind` discriminator and `kind_specific JSONB`. Per DEC-B1.3, IDs are fresh Phoenix CRM IDs with `external_id_map` back-references.

#### Domain breakdown (eight CRM domains + cross-cutting)

1. **Customers** — `customers`, `customer_contacts` (1:N), `customer_locations` (1:N), `customer_tags` (M:N).
2. **Jobs** — `jobs`, `job_appointments` (1:N), `job_techs_assigned` (M:N), `job_notes` (1:N append-only), `job_documents` (1:N), `job_signatures` (1:N), `job_visits` (1:N), `job_statuses` (lookup).
3. **Estimates** — `estimates`, `estimate_line_items` references the polymorphic `line_items` table.
4. **Invoices** — `invoices`, `invoice_line_items` references `line_items`, `invoice_payments` (1:N).
5. **Scheduling** — `appointments`, `technician_availability`, `routes` (with GPS waypoints).
6. **Pricebook** — `services`, `materials`, `equipment`, `categories` (hierarchical via `parent_category_id`), `vendor_prices` (M:N materials × vendors with effective dates), `markup_rules` (Premium / Member / List as configurable rows, not constants).
7. **Reporting** — materialized views: `mv_jobs_by_status_by_week`, `mv_estimate_pipeline`, `mv_invoice_aging`, `mv_tech_utilization`, `mv_revenue_by_source`. Refreshed hourly initially; tune later.
8. **History** — derived from `audit_log` via `entity_history` views projecting "everything that happened to record X."

#### Conventions (apply to every table below)

- `snake_case` table and column names.
- High-volume tables (`audit_log`, `line_items`, `appointments`, `customer_contacts`): `id BIGSERIAL`.
- Cross-system records (anything in `external_id_map`, anything an agent or external system references): `id UUID DEFAULT gen_random_uuid()`.
- All tables: `created_at TIMESTAMPTZ NOT NULL DEFAULT now()`, `updated_at TIMESTAMPTZ NOT NULL DEFAULT now()`, `archived_at TIMESTAMPTZ NULL`, `created_by UUID REFERENCES users(id)`, `updated_by UUID REFERENCES users(id)`. No `DELETE` ever.
- Money: `NUMERIC(12, 2)`. Currency implicit USD; add `currency_code` if Phoenix expands.
- Every domain table has `external_id_map` rows linking it to the SF counterpart (during Phase C and as historical record forever).

#### DDL sketch — core tables

This is the shape for Phase C to start from. Field lists are not exhaustive — they're the ones Phase B has committed to. JSONB columns absorb the SF response shape's long tail.

```sql
-- =========================================================================
-- Identity & cross-cutting
-- =========================================================================

CREATE TABLE users (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  m365_object_id  UUID UNIQUE NOT NULL,        -- Entra ID immutable object ID
  email           CITEXT UNIQUE NOT NULL,
  display_name    TEXT NOT NULL,
  status          TEXT NOT NULL DEFAULT 'active',  -- 'active' | 'inactive'
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  archived_at     TIMESTAMPTZ
);

CREATE TABLE roles (
  id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name      TEXT UNIQUE NOT NULL,         -- 'viewer' | 'editor' | 'finance' | 'admin' | 'agent:<name>'
  is_agent  BOOLEAN NOT NULL DEFAULT false
);

CREATE TABLE permissions (
  id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  resource  TEXT NOT NULL,                -- 'customers' | 'estimates' | 'pricebook' | ...
  action    TEXT NOT NULL,                -- 'read' | 'write' | 'archive' | 'sell' | ...
  UNIQUE (resource, action)
);

CREATE TABLE role_permissions (
  role_id        UUID REFERENCES roles(id),
  permission_id  UUID REFERENCES permissions(id),
  PRIMARY KEY (role_id, permission_id)
);

CREATE TABLE user_roles (
  user_id  UUID REFERENCES users(id),
  role_id  UUID REFERENCES roles(id),
  granted_by UUID REFERENCES users(id),
  granted_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (user_id, role_id)
);

CREATE TABLE external_id_map (
  id              BIGSERIAL PRIMARY KEY,
  local_table     TEXT NOT NULL,           -- 'customers' | 'jobs' | ...
  local_id        TEXT NOT NULL,           -- string for cross-type compat
  external_system TEXT NOT NULL,           -- 'service_fusion' | 'quickbooks' | 'rexel'
  external_id     TEXT NOT NULL,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (local_table, local_id, external_system),
  UNIQUE (external_system, external_id, local_table)
);

CREATE INDEX idx_eim_local  ON external_id_map(local_table, local_id);
CREATE INDEX idx_eim_remote ON external_id_map(external_system, external_id);

CREATE TABLE sync_state (
  id                   UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  external_system      TEXT NOT NULL,
  external_table       TEXT NOT NULL,         -- 'customers' | 'jobs' | ...
  cursor_field         TEXT NOT NULL,         -- 'modified_on' typically
  cursor_value         TIMESTAMPTZ,           -- last-seen value
  last_run_at          TIMESTAMPTZ,
  last_success_at      TIMESTAMPTZ,
  status               TEXT NOT NULL,         -- 'idle' | 'running' | 'failed'
  last_error           TEXT,
  records_seen_total   BIGINT NOT NULL DEFAULT 0,
  UNIQUE (external_system, external_table)
);

CREATE TABLE custom_field_schemas (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  target_table TEXT NOT NULL,             -- 'customers' | 'jobs' | ...
  field_key    TEXT NOT NULL,
  json_type    TEXT NOT NULL,             -- 'string' | 'number' | 'boolean' | 'date' | 'enum' | 'object'
  enum_values  TEXT[],                    -- when json_type='enum'
  required     BOOLEAN NOT NULL DEFAULT false,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  archived_at  TIMESTAMPTZ,
  UNIQUE (target_table, field_key)
);

CREATE TABLE dry_run_previews (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  actor_id          UUID NOT NULL REFERENCES users(id),
  resource          TEXT NOT NULL,        -- 'estimates' | 'customers' | ...
  action            TEXT NOT NULL,        -- 'create' | 'update' | 'archive' | 'sell' | ...
  payload           JSONB NOT NULL,       -- what would be sent
  computed_result   JSONB NOT NULL,       -- what the system would do
  warnings          JSONB,                -- array of warning strings
  required_role     TEXT,
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at        TIMESTAMPTZ NOT NULL,
  consumed_at       TIMESTAMPTZ,          -- set when commit succeeds
  consumed_correlation_id UUID
);

CREATE INDEX idx_previews_actor_active ON dry_run_previews(actor_id, expires_at)
  WHERE consumed_at IS NULL;

CREATE TABLE pending_approvals (
  id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  preview_id          UUID NOT NULL REFERENCES dry_run_previews(id),
  initiator_id        UUID NOT NULL REFERENCES users(id),
  required_role       TEXT NOT NULL,         -- role that must approve
  required_count      SMALLINT NOT NULL DEFAULT 1,
  approver_ids        UUID[] NOT NULL DEFAULT '{}',  -- who has approved so far
  status              TEXT NOT NULL DEFAULT 'pending', -- 'pending' | 'approved' | 'rejected' | 'expired'
  rejection_reason    TEXT,
  notification_channel TEXT NOT NULL,         -- 'teams'
  notification_sent_at TIMESTAMPTZ,
  created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at          TIMESTAMPTZ NOT NULL,
  resolved_at         TIMESTAMPTZ
);

-- =========================================================================
-- Customers domain
-- =========================================================================

CREATE TABLE customers (
  id                 UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  display_name       TEXT NOT NULL,
  customer_type      TEXT NOT NULL,             -- 'Residential' | 'Commercial'
  primary_email      CITEXT,
  primary_phone      TEXT,
  primary_address    JSONB,                     -- street/unit/city/state/zip/country/lat/lng
  custom_fields      JSONB NOT NULL DEFAULT '{}'::jsonb,
  source             TEXT,                      -- maps to SF /v1/sources name
  notes              TEXT,
  is_active          BOOLEAN NOT NULL DEFAULT true,
  membership_tier    TEXT,                      -- 'List' (default) | 'Member' | 'Premium'
  created_at         TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at         TIMESTAMPTZ NOT NULL DEFAULT now(),
  archived_at        TIMESTAMPTZ,
  created_by         UUID REFERENCES users(id),
  updated_by         UUID REFERENCES users(id)
);
CREATE INDEX idx_customers_phone ON customers(primary_phone);
CREATE INDEX idx_customers_email ON customers(primary_email);
CREATE INDEX idx_customers_name_trgm ON customers USING gin (display_name gin_trgm_ops);

CREATE TABLE customer_contacts (
  id            BIGSERIAL PRIMARY KEY,
  customer_id   UUID NOT NULL REFERENCES customers(id),
  kind          TEXT NOT NULL,             -- 'phone' | 'email' | 'fax' | 'mobile'
  value         TEXT NOT NULL,
  memo          TEXT,
  is_primary    BOOLEAN NOT NULL DEFAULT false,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  archived_at   TIMESTAMPTZ
);
CREATE INDEX idx_contacts_customer ON customer_contacts(customer_id) WHERE archived_at IS NULL;

CREATE TABLE customer_locations (
  id            BIGSERIAL PRIMARY KEY,
  customer_id   UUID NOT NULL REFERENCES customers(id),
  label         TEXT,                      -- 'Main', 'Garage', etc.
  address       JSONB NOT NULL,
  is_primary    BOOLEAN NOT NULL DEFAULT false,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  archived_at   TIMESTAMPTZ
);

CREATE TABLE customer_tags (
  customer_id  UUID REFERENCES customers(id),
  tag          TEXT NOT NULL,
  applied_by   UUID REFERENCES users(id),
  applied_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (customer_id, tag)
);

-- =========================================================================
-- Jobs domain
-- =========================================================================

CREATE TABLE job_statuses (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  code        TEXT UNIQUE NOT NULL,        -- '02_SCH', '07_STA', etc. (matches SF codes)
  name        TEXT NOT NULL,
  category    TEXT NOT NULL,               -- 'OPEN' | 'OPEN_ACTIVE' | 'CLOSED'
  sort_order  SMALLINT NOT NULL DEFAULT 0,
  archived_at TIMESTAMPTZ
);

CREATE TABLE jobs (
  id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  job_number          TEXT UNIQUE,                -- SF-assigned during Phase C
  customer_id         UUID NOT NULL REFERENCES customers(id),
```

#### Trigger pattern for domain tables (DEC-B2.1)

Every domain table that holds business data gets a row-level AFTER trigger writing to `audit_log`. The pattern is uniform — `customers` shown as the canonical example; `jobs`, `estimates`, `invoices`, `line_items`, `services`, `materials`, `markup_rules` etc. follow the same pattern.

```sql
CREATE OR REPLACE FUNCTION audit_writer()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
DECLARE
  v_action          TEXT;
  v_before          JSONB;
  v_after           JSONB;
  v_diff            JSONB := '{}'::jsonb;
  v_correlation_id  UUID;
  v_actor_id        UUID;
  v_actor_kind      TEXT;
  v_source          TEXT;
  v_reason          TEXT;
  k                 TEXT;
BEGIN
  -- Caller-supplied context via session GUCs (set at the start of every transaction
  -- by the application/session middleware; missing context is a hard error so we
  -- can't accidentally write unattributed audit rows)
  v_correlation_id := current_setting('phoenix.correlation_id', false)::UUID;
  v_actor_id       := current_setting('phoenix.actor_id', false)::UUID;
  v_actor_kind     := current_setting('phoenix.actor_kind', false);
  v_source         := current_setting('phoenix.source', false);
  v_reason         := current_setting('phoenix.reason', true);  -- nullable

  IF TG_OP = 'INSERT' THEN
    v_action := 'create';
    v_before := NULL;
    v_after  := to_jsonb(NEW);
  ELSIF TG_OP = 'UPDATE' THEN
    -- archive vs update: NEW.archived_at NOT NULL AND OLD.archived_at IS NULL
    IF NEW.archived_at IS NOT NULL AND OLD.archived_at IS NULL THEN
      v_action := 'archive';
    ELSE
      v_action := 'update';
    END IF;
    v_before := to_jsonb(OLD);
    v_after  := to_jsonb(NEW);
    -- Compute diff: keys whose values changed
    FOR k IN SELECT jsonb_object_keys(v_after) LOOP
      IF v_before->k IS DISTINCT FROM v_after->k THEN
        v_diff := v_diff || jsonb_build_object(k, jsonb_build_object('old', v_before->k, 'new', v_after->k));
      END IF;
    END LOOP;
  END IF;

  INSERT INTO audit_log (
    correlation_id, actor_id, actor_kind, action,
    target_system, target_table, target_id,
    before_state, after_state, diff, reason,
    status, source
  ) VALUES (
    v_correlation_id, v_actor_id, v_actor_kind, v_action,
    'phoenix_crm', TG_TABLE_NAME, COALESCE(NEW.id, OLD.id)::TEXT,
    v_before, v_after, v_diff, v_reason,
    'success', v_source
  );

  RETURN NEW;
END
$$;

CREATE TRIGGER audit_customers_iu
  AFTER INSERT OR UPDATE ON customers
  FOR EACH ROW EXECUTE FUNCTION audit_writer();
-- ... same trigger attached to every business table.
```

The session GUCs (`phoenix.correlation_id`, `phoenix.actor_id`, `phoenix.actor_kind`, `phoenix.source`, optional `phoenix.reason`) are set at the start of every DB transaction by the application's connection-pool middleware. The trigger uses `current_setting(..., false)` for required GUCs — if missing, the write **fails loudly** rather than producing an unattributed audit row. This is the non-negotiable #1 enforcement at the storage layer.

External API calls (layer 1 of DEC-11) and business events (layer 3) are inserted into `audit_log` directly by the application — they don't have a domain-table mutation to attach a trigger to. Those INSERTs use the same `audit_log` permissions and same context GUCs.

#### PCS pino-log replay (DEC-B2.3)

At Phase C start, pino write logs from PCS get replayed into `audit_log` so we don't lose historical SF write events. Design:

- **Source.** PCS pino logs from Shane's MacBook (where stdio mode runs) — these go to Console output and possibly a local log file. The `logApproval()` calls (once Phase A wires them up at item 13) emit structured JSON with `type: 'approval'`, `operation`, `approvedBy`, `timestamp`, `details`. Phase A also adds a file-rotation transport (e.g. `pino/file` to `~/.phoenix/pcs/audit-YYYY-MM.log`) so the replay has a stable source.
- **Filter.** Replay **only write events**: `type === 'approval'` rows AND any `tool_execution` rows where the tool name matches `^servicefusion_(create_|update_)` (note: SF v1 has no update path, so `create_*` covers all writes today). Read calls and rate-limit warnings are filtered out — too noisy, no historical value.
- **Retroactive correlation_id assignment.** Pino logs don't carry `correlation_id` until Phase A item 13 wires AsyncLocalStorage propagation. Pre-Phase-A logs get a deterministic synthetic ID per session: `correlation_id = uuid_v5(namespace='pcs-replay', name=f'{session_start_iso}#{event_index}')`. Post-Phase-A logs use their real correlation_id.
- **Mapping.** Each replayed row becomes an `audit_log` INSERT with:
  - `actor_id`: looked up from `users` by email (Shane is the only Phase-A actor); if no match, a synthetic `migration` actor.
  - `actor_kind`: `'migration'` (the replay itself), with the original actor in `reason`.
  - `action`: `'external_call'` (with `request_payload`) and a paired `'external_response'` row.
  - `target_system`: `'service_fusion'`.
  - `source`: `'migration'`.
  - `occurred_at`: original timestamp from the pino row (NOT now()).
- **Tooling.** Replay is a one-shot CLI run from `phoenix-crm` migrations (`pnpm migrate:replay-pcs --from <file>`), idempotent (pre-checks `external_id_map` for replayed event keys to avoid double-insertion).
- **Validation.** After replay, count `audit_log` rows with `source = 'migration'` and confirm against the source pino file's filtered row count.

### B3 — Write Controls

Per DEC-B3.1, second-approver notifications go to Microsoft Teams with magic-link approve/reject. Per DEC-B3.2, preview TTL is 30 minutes default with 5-minute override for bulk/high-impact. Per DEC-B3.3, agents must preview→log→commit with correlation_id chain — no exemption.

#### RBAC model

- **Roles:** `viewer`, `editor`, `finance` (editor + invoice/payment writes), `admin`, plus per-agent roles like `agent:quote-generator`, `agent:pricing-analyst`, `agent:phoenix-courier`, etc.
- **Phoenix Electric humans:** Shane = `admin`; Stephanie = `editor` + `finance`; Ash = `editor`; future office staff per role.
- **Agents:** principle of least privilege per agent. Quote Generator can `estimates:write` but not `estimates:sell`. Pricing Analyst can `pricebook:read` but not `pricebook:write`. Phoenix Courier (Graph read-only) gets no Phoenix CRM write permissions. Each agent's role is scoped to the minimum verbs it needs. (ServiceTitan Director was a previously-listed agent; it is removed per § Removed From Scope.)

#### Dry-run-by-default pattern

Every write tool exposes `preview` and `commit` modes.

```
POST /v1/estimates  { mode: 'preview', ...payload }
→ 200 OK
{
  preview_id: '<uuid>',
  expires_at: '<+30min ISO>',                        // 5min for bulk/high-impact (DEC-B3.2)
  would_create: { id: null, ...computed_estimate },
  would_affect: ['customer:<uuid>', 'job:<uuid>'],
  would_call_external: ['service_fusion: POST /v1/estimates'],   // empty post-cutover
  diff: null,                                        // populated for updates
  warnings: ['Customer is tagged Member (+30%); estimate computed at List (+40%) — confirm intent'],
  required_role: 'editor',
  privileged: false                                  // true → multi-step approval
}

POST /v1/estimates  { mode: 'commit', preview_id: '<uuid>' }
→ 201 Created  (or 202 Accepted, awaiting approval — see below)
```

The commit call MUST reference a valid, non-expired `preview_id` generated by the same actor. Enforced in API middleware against `dry_run_previews`. The `consumed_at` and `consumed_correlation_id` fields prevent replay.

`dry_run_previews.expires_at` is set per-action:

```
default:                    now() + 30 minutes
bulk operations:            now() + 5 minutes      -- e.g. pricebook:bulk_update
high-impact privileged:     now() + 5 minutes      -- e.g. users:manage, estimates:sell
```

#### Multi-step approval — Microsoft Teams flow (DEC-B3.1)

Privileged actions (`estimates:sell`, `pricebook:bulk_update` over N items, `users:manage`, any reconciliation conflict resolution) require a second human's approval before commit.

**Flow:**

1. Actor calls `commit` with `preview_id`. Server detects `privileged: true`, returns `202 Accepted, awaiting_approval`. A `pending_approvals` row is created (`status='pending'`, `expires_at = now() + 24h`).
2. Server posts to a Microsoft Teams channel (configured per Phoenix tenant — likely `#phoenix-approvals`) using Graph API `chatMessage` with an Adaptive Card. The card shows:
   - Initiator's name + avatar (from M365 SSO).
   - Action summary ("Sell estimate #1234 for $4,500 to Smith Residence").
   - Link to a full preview view in Phoenix CRM web UI.
   - Two buttons: **Approve** and **Reject** (with required reason).
3. Buttons are signed magic links: `https://phoenix-crm.<domain>/approve/<jwt>`. The JWT payload includes `pending_approval_id`, `expires_at` (matches the `pending_approvals.expires_at`), and is signed by the Phoenix CRM API server with a key from Key Vault.
4. Approver clicks. Phoenix CRM web UI authenticates the click via M365 SSO. The approver's identity (M365 object ID) must:
   - Match a `users` row.
   - Hold the `required_role` for this approval.
   - **Not be the initiator** (no self-approval, even if initiator is `admin`).
5. On approval: `pending_approvals.status = 'approved'`, `approver_ids` appended; if `required_count` met, the original action commits (the deferred work runs server-side, identical to the non-privileged path), `audit_log` shows both the initiator's `commit` action and the approver's `approve` business event with same `correlation_id`.
6. On rejection: `pending_approvals.status = 'rejected'`, `rejection_reason` stored, action does NOT commit, initiator notified via Teams DM (or email fallback).
7. On expiry (24h): `pending_approvals.status = 'expired'`, action does NOT commit. The `dry_run_previews` row stays consumed so the same preview can't be retried; initiator must preview again.

**Approver identification:** M365 SSO is the only auth path (DEC-5). The JWT's `sub` claim is the M365 object ID; the click-time M365 SSO sign-in confirms it's actually that human at the keyboard (or someone with their session — out of scope; Conditional Access policies on the M365 tenant are the second line of defense).

**Why Teams over email:** Phoenix already has Teams. Email magic-links are phishable (a Teams magic-link inside a Microsoft Adaptive Card with the initiator's avatar is harder to spoof). Telegram is fine for Shane personally but doesn't extend to Stephanie/Ash without giving them a Telegram account they don't otherwise need.

#### Agents — preview→log→commit with correlation_id chain (DEC-B3.3)

Programmatic actors (agents) follow the same dry-run pattern as humans, no exemption. The pattern:

```
1. Agent assembles inputs → sets phoenix.correlation_id = <new uuid>.
2. Agent calls preview endpoint with payload + correlation_id.
   Server returns preview_id, persists to dry_run_previews with that correlation_id.
3. Agent logs the preview result locally and continues its reasoning.
   If the agent decides not to commit, the preview expires harmlessly.
4. Agent calls commit endpoint with preview_id and SAME correlation_id.
   Server verifies (preview_id, actor_id) match; verifies correlation_id of the commit
   request matches the preview's correlation_id. Mismatch = 409 Conflict.
5. Commit succeeds. audit_log shows two rows: the preview as a 'business_event' (kind='preview_generated')
   and the commit as 'create'/'update'/'archive', both linked by correlation_id.
```

The correlation_id chain is the audit thread. A forensic question like "the system created this customer last Tuesday — what reasoning produced that?" is answered by `SELECT * FROM audit_log WHERE correlation_id = (SELECT correlation_id FROM audit_log WHERE target_table='customers' AND target_id=<uuid> ORDER BY occurred_at LIMIT 1)`. That returns: the preview, the agent's `reason` field, the commit, and any external API calls the commit triggered — all in time order.

The "deterministic agent inputs don't need preview" argument is rejected because deterministic inputs are exactly where bypasses ossify into "we never need preview here" and then the day a non-deterministic edge case happens there's no preview record to investigate.

- `invoices:void` (financial)

### B5 — Migration Path

Per DEC-2 read-only mirror with writes-target-SF-only during Phase C. Per DEC-9 rolling backfill into Phoenix CRM AND SF read-only forever post-cutover. Per DEC-B5.3 per-table sync cursors (already shown as `sync_state` in B1 DDL). Per DEC-B5.2 CSV export from SF admin for one-time historical archive of API-unreachable data. #### Mirror strategy (Phase C)

- **Initial backfill.** One-time full pull of every record from every SF endpoint PCS exposes. Stored in Phoenix CRM tables with `external_id_map` rows. Migration tool only — not the normal app path. `actor_kind = 'migration'` for all backfill audit rows.
- **Incremental sync.** Every 15 min (initial; tune later), each SF table polled with its own cursor (`sync_state.cursor_value` = last-seen `modified_on`). Upsert changes; insert audit rows with `actor_kind = 'system'`, `source = 'cron'`. Conflicts impossible — Phase C doesn't accept Phoenix-CRM-side writes.
- **Sync state.** `sync_state` table per (external_system, external_table). Health endpoint exposes lag per table.
- **Failure modes.** Sync stale > 1h = Yellow event (Teams notification). Stale > 4h = Red (Shane direct). Both trigger an `audit_log` business event.

#### CSV-export plan for one-time historical archive (DEC-B5.2)

SF web UI exports several record types as CSV that aren't available via API: telecom call logs, dispatch-board state snapshots, certain custom-field histories, web-UI-only computed fields. Plan:

1. Shane runs SF admin's CSV export for each available export type once, near Slice 1 backfill time. Files go into Azure Blob Storage at `phoenix-crm-historical/sf-csv-export/<YYYY-MM-DD>/<export-type>.csv`.
2. A separate `pnpm migrate:ingest-sf-csv --type <export-type> --file <blob-path>` tool reads each CSV, maps to Phoenix CRM tables (or to a holding table `historical_sf_<export-type>` if no Phoenix CRM home exists), writes `external_id_map` rows, and inserts `audit_log` rows with `actor_kind='migration'`, `source='migration'`, `reason='SF CSV export <date>'`.
3. The CSV files themselves are kept in blob storage forever (versioned). Phoenix CRM doesn't need to re-ingest, but having the originals means a future schema change can re-derive without going back to Shane.
4. Export types we know we need: `calls`, `dispatch-state`, `customer-custom-field-history`. Phase B v2 still needs Shane's input on the full export-type list — see § What Phase B v2 still needs from Shane.

What we explicitly do NOT recover: telecom audio recordings (separate hosting in SF, not in CSV export), dispatch board's exact pixel state (not meaningful data), web-UI-only fields that have no schema (we live without).

#### Conflict resolution: not applicable in Phase C

Per DEC-2 Phoenix CRM doesn't accept its own writes during Phase C. SF is the only writer. Phase D cutover makes SF read-only and Phoenix CRM the only writer. There is never a window with two concurrent writers, so there is never a conflict to resolve.

The `pending_approvals` "reconciliation conflict" privileged action exists for one specific case: nightly drift detection finds a mismatch between SF and Phoenix CRM, and resolution requires choosing which is correct. That's a privileged action requiring two admins.

#### Cutover (Phase D — outline; detail in Phase D section)

- SF read-only at T-0 (revoke OAuth client_secret in SF admin).
- Phoenix CRM write paths flip from "reject" to "accept."
- Final reconciliation pass confirms parity.
- "Writes-disabled in SF" verification by attempting a write and confirming SF rejects it.

#### Rollback story

- **During Phase C:** nothing to roll back; SF is SoR, Phoenix CRM is a derived view.
- **At cutover:** restore SF OAuth client_secret, flip Phoenix CRM writes back to "reject," reverse-apply any Phoenix-CRM-only writes from the cutover window manually with two-admin approval.
- **Post-cutover:** SF stays read-only forever (DEC-9). Lost-record recovery: SF has it.

## Phase D — Cutover

**Status:** Blocked on Phase C completion (90 days clean across all six slices).

**Goal:** Define what "Phoenix CRM is proven" means and the procedure for flipping SF from system-of-record to read-only archive.

This phase is design-only at this point. The actual cutover plan is written in detail just before cutover, when we know what the system actually looks like. What follows is acceptance criteria and procedure outline.

### "Proven" criteria (all must hold)

1. **Functional completeness.** Every B4 flow works end-to-end in Phoenix CRM. Stephanie has gone an entire week using only Phoenix CRM (with SF tab closed) and reports no missing capabilities.
2. **Data parity.** Nightly drift detection < 0.05% for 60 consecutive days. Random spot-check of 100 records each in customers, jobs, estimates, invoices: all fields match.
3. **Audit trail proven useful.** A real forensics question has been answered using `audit_log` queries. The infrastructure is exercised, not theoretical.
4. **Performance under realistic load.** 100 concurrent users sustained for 1 hour without degradation.
5. **Backup and restore tested.** Postgres backup taken, restored to fresh instance, verified. Audit log preserved through restore.
6. **Security review passed.** External or internal review of RBAC, write controls, secrets handling, audit-trail integrity. Not a checkbox — a real review with findings addressed.
7. **Stephanie, Ash, and Shane all sign off.** Independently. If any of them isn't ready, cutover doesn't happen.
8. **Field-tech-tooling design exists** (per DEC-10, post-cutover, but design must exist so techs aren't surprised).

### Cutover procedure (outline — detail at cutover-time)

**T-7 days — Communication.** Notify Phoenix Electric staff. Final dry-run cutover in staging.

**T-1 day — Pre-flight.** Last full sync. Drift check (zero, not "low"). Backup Phoenix CRM database. Backup SF data via final CSV export.

**T-0 — Cutover** (target window: Friday evening, low-activity):

1. Announce maintenance window (30 min planned).
2. Stop incremental sync.
3. Run final delta sync.
4. Drift check. Zero required.
5. Revoke SF OAuth client_secret in SF admin (disables PCS writes; reads will fail too — intentional).
6. Flip Phoenix CRM write paths from "reject" to "accept."
7. Update agent config: Phoenix Courier, Finance Analyst, Pricing Analyst, Quote Generator, Schedule Coordinator, Security Sentinel, Knowledge Keeper, Code Reviewer, Compliance Auditor — all hit Phoenix CRM endpoints. All live agents re-pointed. ServiceTitan Director is not in this list (removed from scope; see § Removed From Scope).
8. PCS stays installed; `ALLOW_SF_WRITES` hardcoded to false; plugin remains a read-only window into the SF historical archive.
9. Monitor 4 hours. Anything weird → Rollback.
10. Announce completion.

**T+1 day.** Daily standup focused on cutover issues. Stephanie's morning workflow under the new system. Shane reviews `audit_log` volume and content.

**T+7 days.** Retrospective. Updates to runbook.

### Rollback (if Phase D fails within 48 hours)

1. Restore SF OAuth client_secret. SF write path through PCS resumes.
2. Flip Phoenix CRM write paths back to "reject." Phoenix CRM resumes mirror mode.
3. Diff writes that landed in Phoenix CRM during cutover window. Reverse-apply as SF writes manually with two-admin approval (per non-negotiable #3 — these are retroactive writes).
4. Schedule second cutover only after blocking issue fixed AND new dry-run passes.

After 48h rollback gets harder; after 7 days it's effectively "double-write to converge" which is its own design. The 48-hour window is the safety zone.

### Phase D deferred decisions, now declarative

- **SF subscription post-cutover (DEC-D1):** keep cheapest SF tier indefinitely (read-only API + CSV export). Cheap rollback insurance.
- **PCS / `phoenix-current-software` repo post-cutover (DEC-D2):** archive read-only. Never deleted. MCP server still functions as historical SF read window.
- **Agents post-cutover (DEC-D3 — now OBSOLETE for ServiceTitan; the rest of the answer stands):** ALL live agents re-pointed at Phoenix CRM. SF queries via PCS become read-only fallback for reconciliation forensics only. ServiceTitan Director does not exist in the post-cutover plane (or any other Phase B/C/D plane).

### Phase D dependencies

- Phase C complete with 90 days clean.
- All eight proven-criteria items above.
- Shane's calendar: cutover takes 30 min planned + 4 hours monitoring; not during active business events.
